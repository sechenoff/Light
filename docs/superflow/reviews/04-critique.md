# Финансы: критика спецификации (Agent 4)

## TL;DR

1. **Phase 1 «без schema» — это иллюзия.** Половина F-фиксов в Phase 1 опирается на канонический термин «Срок оплаты» = `Invoice.dueDate`, которого ещё нет. F4 (mêртвый `expectedPaymentDate`) НЕ закрывается без модели Invoice — Agent 3 в Phase 1 пишет «period-selector рабочий», но не отвечает, как наполняются aging buckets без invoice.dueDate. Phase 1 рискует быть косметикой.
2. **Backfill историчных броней в Invoice — данные-токсичная миграция.** Spec предлагает «один FULL-invoice per Booking с finalAmount, dueDate = endDate + 7d». Это **подделка финансового документа** для броней, по которым реальный счёт никто не выставлял. После применения — aging заполнится фантомными «просрочками», бухгалтер увидит ИНН/реквизиты на счёте, который клиент никогда не получал.
3. **Нумерация счетов не специфицирована.** «СЧ-2026-0042» — пример, без правил гарантии непрерывности, годового сброса, синхронизации с 1С / Тинькофф. Это **юридический риск**: ФНС при проверке не любит дыры в нумерации.
4. **WAREHOUSE «без меню, но с кнопкой» — leaky abstraction.** Spec расширяет роль WH (запись платежа, видимость финансовой секции на броне, видимость PDF счёта) и при этом «не расширяет права». Это противоречит self-описанному принципу Out-of-scope #1 («Roles/permissions changes»). По факту меняется матрица прав: WH впервые получает write-доступ к финансам.
5. **Cancel/refund flow для предоплаты — не описан.** Самый частый кейс: клиент внёс депозит, отменил съёмку. Где живёт «удержать комиссию», где «вернуть полностью»? В spec упомянут Refund как примитив, но lifecycle «отмена брони с депозитом» не описан, в edge cases отсутствует.

## Endorsements (что Agent 3 сделал правильно)

1. **«Booking — single source of truth»** — правильный архитектурный сдвиг; снимает класс багов из A3 + F1.
2. **Иммутабельный платёж + Refund как отдельная запись (P12)** — золотой стандарт, фиксирует A2 anti-pattern.
3. **Разделение Booking.status и финансового статуса (A6)** — отказ от введения CLOSED через двойной StatusPill — здравая прагматика.
4. **Один `RecordPaymentModal` для трёх точек входа** — закрывает F1/F2 элегантно, без расширения меню для WH.
5. **Aging buckets с цветным светофором (P2)** — must-have, индустрия-консенсус, корректно перенесено.
6. **Удаление delete-а платежа из UI (F9)** — закрывает audit-дыру, согласуется с уже сделанным rejectBooking-паттерном.
7. **Терминологический glossary с фиксацией грамматического рода** — детальная работа, ловит реальные расхождения из Agent 1.
8. **Forecast как детерминированная сумма confirmed.unpaid** — отказ от ML, прагматично и отвечает реальному размеру (50–200 броней/мес).
9. **Phase 3 как parking-lot** — фискал/УПД/email-напоминания корректно отложены, не раздувает scope.
10. **Mobile-first для записи оплаты на сканере** — попадает в существующий warehouse-flow без второго UI.

## Challenges (что нужно пересмотреть)

### Challenge 1: Phase 1 не закрывает то, что заявлено
**Decision:** Spec говорит «Phase 1 — quick wins без schema-изменений», и в Phase 1 указаны F4, F5, F11, F12 fix-ы.
**Problem:** F4 (мёртвый `expectedPaymentDate`) явно не fix-ится без UI ввода в `/bookings/new` или `/bookings/[id]/edit`. Spec говорит «soft-deprecate в Phase 2» — то есть в Phase 1 поле остаётся пустым. Aging buckets в Phase 1 будут такими же пустыми, как сегодня. Aging таблица «компактная» в `/finance` (Phase 1) физически не имеет данных.
**Impact:** После Phase 1 пользователь увидит «новый» `/finance` с теми же пустыми bucket'ами. Восприятие — «они переименовали и покрасили, но ничего не починили». Эрозия доверия к редизайну.
**Alternative:** Либо (a) добавить в Phase 1 поле «Срок оплаты» в форму брони (минимальное schema-нейтральное изменение — поле уже есть в БД), либо (b) явно сказать «aging переезжает в Phase 2», и в Phase 1 на `/finance/debts` остаётся текущая логика без bucket-ов.
**Severity:** HIGH

### Challenge 2: Backfill «один FULL-invoice per Booking» искажает данные
**Decision:** «для каждой `Booking` с `finalAmount` и `paymentStatus != PAID` создаём один `Invoice { kind: FULL, amount: finalAmount, dueDate: booking.endDate + 7d, status: derived }`».
**Problem:** Это создаёт инвойс с `issuedAt`/`number`, у которого нет реального документального следа. Если потом подключить PDF-генерацию счёта с реквизитами — у этих исторических invoice реквизитов нет, нумерация подделана, дата выставления = дата миграции (а не реальная). Бухгалтерская сверка с 1С/Тинькофф ломается.
**Impact:** ФНС-риск (фиктивные счета в системе), некорректный aging (просрочка считается с фейковой dueDate), невозможность ретроактивно «выставить» исторический invoice — он уже существует.
**Alternative:** Backfill только `Invoice` со статусом VOIDED или новым статусом `LEGACY` без `number`/`issuedAt`. Либо: Phase 2 запускается «с чистого листа» — старые брони продолжают работать через `Booking.amountOutstanding` legacy, новые брони получают invoice. Дуальный режим на 6 месяцев, потом cleanup.
**Severity:** HIGH

### Challenge 3: Нумерация счетов не определена
**Decision:** `Invoice.number @unique` с примером «СЧ-2026-0042».
**Problem:** Spec не определяет: (a) кто ведёт счётчик (БД-триггер? application-level mutex?), (b) гарантия непрерывности (ФНС не любит дыры — а void-нутый счёт оставляет дыру в номере), (c) сброс по году, (d) синхронизация с внешней нумерацией (если бухгалтер ведёт параллельную нумерацию в 1С).
**Impact:** Race condition при concurrent issue → дублирование номера → unique-constraint failure → 500. Юр. риск при проверке. Невозможность ретроактивно перенумеровать.
**Alternative:** Указать: (a) счётчик в transaction-locked sequence, (b) void = сохраняет номер (видимо как «СЧ-2026-0042 АННУЛИРОВАН»), (c) формат настраивается в `/settings/organization`, (d) явное указание «нумерация Light Rental ≠ нумерация бухгалтерии 1С — это операционные счета».
**Severity:** HIGH

### Challenge 4: WAREHOUSE leaky boundary — это privilege expansion
**Decision:** «WAREHOUSE финансового меню не получает, но получает точечный доступ к одной операции».
**Problem:** Из out-of-scope (10 пунктов) явно нет roles/permissions, но вся spec вертится вокруг расширения WH-роли: запись платежа на брони + на сканере + видимость финансовой секции с suммой/статусом счёта/PDF счёта. Это **повышение прав**, замаскированное под «UI fix». Ранее WH не мог ничего финансового делать (F2 — кнопка молча проваливалась с 403). Теперь WH может писать наличный платёж по любой ISSUED/RETURNED брони.
**Impact:** Регрессия безопасности, не описанная как security change. Не закрыт сценарий: WH с iPhone из дома может «принять оплату» по чужой брони, не отдавая нал. Нет лимита по сумме, нет 4-eye на крупные платежи. Audit есть, но не превентивный контроль.
**Alternative:** Явно вынести «WAREHOUSE может писать платёж» в раздел **«Изменения в правах ролей»** spec'а. Добавить лимит «WH может писать только cash/card, до 100 000 ₽; больше — требует SA approval». Зафиксировать в матрице прав в CLAUDE.md.
**Severity:** HIGH

### Challenge 5: Cancel-with-deposit edge case не описан
**Decision:** Spec вводит Refund как примитив (P12), но lifecycle «бронь отменена после депозита» — нет.
**Problem:** Это самый частый кейс российского кинопроката: клиент внёс 30%, потом проект отменили. Возможные исходы: (a) полный возврат, (b) удержание комиссии (10–30%), (c) перенос депозита на другую бронь, (d) кредит-нота клиенту. Spec молчит.
**Impact:** Реальный пользователь — founder этого rental house — столкнётся с этим в первую же неделю. Без UX-flow он будет делать это «через void + ручной возврат», что снова возвращает нас в anti-pattern A2 (mutable mess).
**Alternative:** Добавить в spec явную секцию «Отмена брони с предоплатой»: при `Booking → CANCELLED` со существующим `Payment.amount > 0` — обязательный диалог «Возврат: полный / частичный / удержание / перенос». Каждый исход — отдельная audit-запись.
**Severity:** HIGH

### Challenge 6: Period-selector через `?period=` упрощение
**Decision:** F5 fix — «period-selector через query param».
**Problem:** Spec описывает «Сегодня / Неделя / Месяц / Квартал / Год / Кастом», но не определяет: (a) Кастом = две даты от/до? Где UI? (b) Что значит «Неделя» — текущая ISO неделя или последние 7 дней? (c) TZ — Москва (как в moscowDate.ts) или UTC? (d) Период применяется к Payment.receivedAt или Invoice.dueDate?
**Impact:** Разные KPI на одной странице будут показывать разные числа за «Месяц», если один считает по received, другой по issued. Возврат F6/F7 (двойная правда) другим путём.
**Alternative:** В spec фиксировать: «Период применяется к Payment.receivedAt для KPI «Получено», к Invoice.dueDate для KPI «Ожидается», к Expense.date для «Расходов». Все периоды в Moscow TZ, см. `moscowDate.ts`».
**Severity:** MEDIUM

### Challenge 7: «PDF счёт = расширение smetaExport» — неверная аналогия
**Decision:** «PDF Счёт — расширение `smetaExport` под новый шаблон».
**Problem:** Смета и счёт юридически разные документы. Смета — internal расчёт, счёт — обязательство клиента заплатить. Шапка, реквизиты, основания, формулы итогов — другие. Расширение `smetaExport` приведёт к тому, что один и тот же модуль рендерит два юр. разных документа.
**Impact:** Через 3 месяца понадобится Акт оказанных услуг — а `smetaExport` уже хост двух разных шаблонов. Через 6 — УПД. Через год — это monolithic dump, поддержка проседает.
**Alternative:** Создать `apps/api/src/services/documentExport/` с подпапками `estimate/`, `invoice/`, `act/`. Общая Cyrillic font logic выносится в shared util. См. как сделан `smetaExport/renderPdf` уже сегодня — это именно тот рефакторинг, который не сделан.
**Severity:** MEDIUM

### Challenge 8: Booking-page-first vs `/finance`-first — для founder неверный порядок
**Decision:** «`/bookings/[id]` — главный финансовый экран, `/finance/*` — агрегация».
**Problem:** Я не оспариваю принцип в целом, но spec плохо отвечает на вопрос «как founder в 18:00 в среду быстро узнаёт “у меня сегодня долг”». В spec он попадает на `/finance` (KPI), затем `/finance/debts` (aging), затем drill в бронь. Это 3 экрана. На `/day` есть KPI «Долги» — клик ведёт «куда?» (F8 закрыт `<Link>`-ом, но не сказано — куда). Если на `/finance/debts` — почему именно туда, а не на `/finance/invoices?status=overdue`? Spec плодит дубли (debts vs invoices?status=overdue — это одни и те же данные).
**Impact:** Информационная архитектура из 6 страниц финансов (`/finance`, `/invoices`, `/payments`, `/debts`, `/expenses`, `/forecast`) — слишком много для shop с 1 пользователем-SA. Пятёрка лучших SaaS, которые цитирует Agent 2, не делают столько подстраниц финансов.
**Alternative:** Урезать до 3 экранов: `/finance` (dashboard с aging, KPI, action list), `/finance/transactions` (объединённый список инвойсов+платежей+расходов с табами и фильтрами), `/finance/expenses` (отдельно из-за approve flow). Forecast — виджет на dashboard. Debts — фильтр на transactions.
**Severity:** MEDIUM

### Challenge 9: Терминологическая миграция — недооценённая стоимость
**Decision:** Глобальная замена 12 терминов в 6 файлах web.
**Problem:** Spec говорит «глобальный поиск/замена + ручная проверка». Но:
- AuditEntry уже содержит strings типа «Заработал», «Платёж создан» — backfill старых записей? нет;
- мокапы в `docs/mockups/` (которые user считает source-of-truth) могут использовать старые термины — sync'аются вручную?
- bot и API возвращают тексты на старых терминах в ответах — миграция или нет?
- ApprovalTimeline и DayWidgets уже шипнуты с старой терминологией.
**Impact:** Несогласованность UI после Phase 1: новый /finance говорит «Получено», бот говорит «выручка», audit говорит «Заработал». Хуже, чем было.
**Alternative:** Терминологический pass должен быть atomic PR со скриптом синхронизации. Указать в spec: «термины трогаются только в UI web; bot и audit-history остаются с историческими, новые записи на новых терминах». Plus: явный glossary в `apps/web/src/lib/financeTerms.ts`.
**Severity:** MEDIUM

### Challenge 10: `voidedAt` semantics в Phase 1 vs Phase 2 — токсичный лаг
**Decision:** «В Phase 1 кнопка “Удалить” заменяется на “Аннулировать (void)” с обязательным reason. Под капотом в Phase 1 это всё ещё `paymentService.deletePayment`».
**Problem:** UI говорит «аннулирован», БД физически удаляет. Audit-запись пишется, но Payment-record — нет. Если в Phase 2 ввести `voidedAt` — старые void'нутые платежи Phase 1 не существуют как записи (delete сработал). Нельзя пересчитать историю.
**Impact:** Финансовая история двух недель Phase 1 не подлежит восстановлению. Отчёты «сколько void за период» не считаются за этот период.
**Alternative:** Не делать «UI говорит одно, БД делает другое». Либо void в Phase 1 = soft-delete с `deletedAt` колонкой (минимальная schema-добавка), либо отложить void-функционал до Phase 2 и в Phase 1 просто скрыть «Удалить».
**Severity:** MEDIUM

### Challenge 11: «Связаться» кнопка — phase split осмысленности нет
**Decision:** Phase 1 — кнопка «Связаться» открывает контакт; Phase 3 — реальная отправка.
**Problem:** «Открывает контакт» — это что? `tel:` ссылка? Открытие client-detail модалки? Spec не уточняет. Если просто показ phone — это меньше, чем то, что уже есть в `/finance/debts` (там можно ткнуть в карточку клиента).
**Impact:** Реализация в Phase 1 может оказаться dummy + текущая Friction 6 не закрыта.
**Alternative:** Конкретизировать: Phase 1 = `tel:` + `mailto:` ссылки в строке, отправляемые через системный обработчик. Phase 3 = email-template с шаблоном напоминания + история отправок.
**Severity:** LOW

### Challenge 12: «Сегодня сделать» action list дублирует /day
**Decision:** На `/finance` секция «Сегодня сделать»: счета к выставлению, просроченные, платежи без привязки.
**Problem:** На `/day` уже есть `DayAlert` (pending-approvals) и `DayTasksWidget` (myTasks). Spec говорит «не дублировать с /day», но содержательно — это вторая action-list на главной финансовой странице. Founder утром идёт на /day, видит alert — потом на /finance, видит second action list. Где правда?
**Impact:** Два списка «что делать» — психологически тревожнее, чем один. Founder не знает, что уже сделал.
**Alternative:** Action items финансовые поднимаются на `/day` для SA как отдельный alert/widget. На `/finance` — только KPI + aging, без action list. Либо наоборот — на /day остаётся только operational, на /finance переносим всё финансовое actionable.
**Severity:** LOW

### Challenge 13: Mobile reality — не walked through
**Decision:** «Mobile-first для записи и пометки оплаты».
**Problem:** Spec не делает walkthrough на 390px. Финансовая секция на `/bookings/[id]`: шапка с двумя StatusPill, сумма-сводка, секция «Счета» (таблица), «Хронология денег» (список), «Связанные расходы». На 390px это легко 2000+px вертикального scroll до того, как WH доберётся до кнопки «Записать оплату».
**Impact:** WAREHOUSE на телефоне теряет primary action. Spec обещает sticky-bottom CTA, но не описывает где.
**Alternative:** Указать в spec: «На mobile (≤640px) на `/bookings/[id]` для роли WH/SA primary action “Записать оплату” — sticky bottom button. Финансовая секция collapsed по умолчанию, только сумма-сводка видна сразу.»
**Severity:** MEDIUM

### Challenge 14: «Гарантированный pipeline» = sum confirmed.unpaid — не отражает реальность
**Decision:** Forecast: «Гарантированный pipeline = sum(confirmed.future.unpaid)».
**Problem:** Confirmed бронь может cancel'нуться, может быть rejected (но это уже рассмотрено), клиент может задержать на 30 дней, может вообще не заплатить. Называть это «гарантированным» — обещание, которое не выполнится.
**Impact:** Founder верит цифре, делает закупку оборудования, цифра не реализуется — недоверие к системе на год.
**Alternative:** Назвать «Подтверждённый pipeline» (а не гарантированный). Добавить рядом «Реализуемость по истории: 87%» — historical conversion из confirmed.unpaid в paid за последние 6 месяцев. Это есть данные, можно посчитать.
**Severity:** LOW

### Challenge 15: Q1 «Invoice обязательный для всех новых» vs «один FULL-invoice» — как UX
**Decision:** Q1: «да, всегда хотя бы один Invoice (FULL по умолчанию). Без invoice не может быть payment.»
**Problem:** Это противоречит ранее заявленной «гибкости». Если invoice автоматический по template (FULL или 30+70), то для founder, у которого 80% клиентов платят 100% по факту — invoice — это лишняя сущность. Сейчас он делает: бронь подтверждена → выдал → получил нал → закрыл. После: бронь подтверждена → автоматом сгенерился DRAFT-invoice → нужно его «выставить» (DRAFT→ISSUED) → получил платёж → invoice-paid.
**Impact:** В простом кейсе (75% случаев) добавляется 1 click («Выставить счёт») без add value. Founder скажет «зачем ты усложнил».
**Alternative:** Auto-issue на approve для default-template. Тогда: бронь approve → invoice сразу ISSUED → payment как сейчас. DRAFT-state имеет смысл только для template «30+70» (deposit ISSUED сразу, final DRAFT до отгрузки).
**Severity:** MEDIUM

## Reframes (что решает не ту задачу)

### Reframe 1: Корневая проблема не «нет Invoice», а «нет UI для expectedPaymentDate»
F4 — мёртвое поле — сегодня уже есть в схеме. Самый дешёвый wedge: добавить datetime-input в `/bookings/new` и `/bookings/[id]/edit`, и aging-buckets начнут работать через 1 день. **Это даст 60% value Invoice-модели за 5% усилий.** Invoice как полноценная модель — нужен только для split-payments (P1) и refund-первого класса. Spec прав в том, что Invoice — правильная конечная цель, но wedge-стратегия (что выкатить за неделю) могла бы быть «expectedPaymentDate в UI + aging buckets + переименование терминов» — без миграций.

### Reframe 2: Главная финансовая боль founder — не «много экранов», а «чек/акт для бухгалтера клиента»
Из CLAUDE.md и комментария user'а — текущая кинопрокатная отрасль работает на счёт+акт+УПД. Spec эти документы откладывает в Phase 3. Но именно отсутствие этих документов заставляет founder вручную делать счёт в Word. Если выбирать одну вещь, которая снимет реальную ежедневную боль — это **PDF Счёт** + **PDF Акт** в Phase 1, до всякого Invoice-модели. Текущий smetaExport умеет рендерить Cyrillic — расширить шаблон до Счёта на оплату это 1–2 дня, не неделя.

### Reframe 3: WAREHOUSE flow «принять нал при выдаче» — не финансовый редизайн, а warehouse-feature
Самая критичная friction (F2) — кладовщик не может зафиксировать наличный платёж при возврате. Это решается одной кнопкой на `/warehouse/scan` после успешного return. **Это feature warehouse-модуля, не finance-редизайна.** Spec правильно её включает, но семантически она могла быть отдельным мини-PR за день, не ждать 4-недельного финансового редизайна.

## Edge cases — что Agent 3 не описал

1. **Бронь CONFIRMED с депозитом → CANCELLED** (см. Challenge 5). Что с деньгами? Кто решает refund vs hold?
2. **Платёж записан на неправильную бронь.** Сейчас через delete + create. После редизайна — void + create на правильной броне? Что с audit? Кому видно «эту опечатку поправили»?
3. **Два платежа в один день: 50k нал + 30k card.** Сводка кассы в конце дня — два разных chip-total'а. Если client потом скажет «верни 20k налом» — refund на какой payment'е, на каком method'е?
4. **Refund: full vs partial vs no-refund-fee.** Spec упомянул refund, но не описал лестницу: возврат 100%, возврат с удержанием X%, возврат на другой method (получили нал — вернули переводом).
5. **Currency.** Spec говорит «100% RUB», но есть кейс — иностранный продакшн платит в USD/EUR. Просто не поддерживать? Тогда explicitly: «многовалютность — out of scope, при попытке создать invoice в не-RUB — 400».
6. **Late return → дополнительная аренда.** Если клиент вернул на 2 дня позже, тариф добавляется. Это новый invoice.kind=CORRECTION? Или amendment к существующему FINAL-invoice?
7. **Discount после issue invoice.** Founder выставил счёт, потом дал скидку. Void+reissue или credit note? Spec упоминает CORRECTION-kind, но не описывает workflow.
8. **Bank wire в пути.** Клиент отправил перевод, в системе ещё нет. Что записать? «Pending payment»? Spec убрал Payment.PENDING как сущность, но реальность подразумевает «обещание заплатить». Куда?
9. **Налоговая запрашивает выписку за период.** Phase 1+2 не описывают экспорт финансовой выписки в стандартном формате (xlsx, csv). XLSX-экспорт упомянут как кнопка, но формат — не описан. ФНС требует специфический формат.
10. **Concurrent issue двух invoice через два устройства.** Race condition на `Invoice.number`. Spec не описывает (см. Challenge 3).
11. **Booking финал-invoice — но пользователь хочет добавить ещё депозит-invoice задним числом.** Spec не описывает — invoice'ов всегда два или один. Что если 2 депозита (предоплата 30%, потом ещё 30% за оборудование)?
12. **Удаление AdminUser, который создал Invoice/Payment.** FK на createdBy — Restrict? Set null? Spec не указывает; в P2003 уже есть прецедент `ADMIN_HAS_AUDIT_HISTORY`.

## Голос пользователя — 5 цитат от лица founder'а

1. *«Зачем мне ещё одна сущность “счёт”? У меня бронь подтверждена — мне нужно знать, оплатил ли клиент. Раньше я смотрел один статус. Теперь смотрю два — статус брони и статус счёта. Вы добавили работы, не добавив value.»*

2. *«Подождите, в первую неделю вы что фиксите? Поле даты на бронях? У меня их 200 — вы хотите, чтобы я заполнил вручную? Или вы засунете туда `endDate + 7d` автоматом? Но это же не дата платежа, а тыкание пальцем в небо.»*

3. *«Кладовщик теперь может принять нал по любой моей броне с телефона. У меня кладовщик неделю как, а вы ему дали ключи от кассы. Где лимит? Где двойное подтверждение?»*

4. *«Счёт без реквизитов — это не счёт. Если PDF в Phase 2 — что я печатаю Phase 1? Смету? Я и так это делаю. Вы ничего не дали, переименовали кнопки.»*

5. *«У меня клиент отменил съёмку через час после внесения 50%. По факту я ему отдаю обратно 35k (15 удерживаю комиссию). В вашей системе это как? Refund на 35? А оставшиеся 15 — это что? Я не вижу этого ни в Phase 1, ни в Phase 2.»*

### 4. УТОЧНИТЬ перед финализацией

1. **Lifecycle отмены брони с деньгами** — workflow + UI.
2. **«Связаться» — это что конкретно** (Challenge 11): `tel:` + `mailto:` или модалка контакта?

4. **PDF Счёт vs Смета — какой документ клиент получает в Phase 1?** Если ничего — то «Phase 1 не даёт user value по мнению founder'а».
5. **Audit-actions list** — для Invoice ровно какие? `INVOICE_CREATED`, `_ISSUED`, `_VOIDED`, `_DUE_DATE_CHANGED`? Spec вскользь упоминает три.
6. **Фильтр /bookings: «активные деньги» (RETURNED + outstanding > 0)** — UI-точка фильтра, какой URL-param.
7. **Backfill: что с CANCELLED бронями?** Spec говорит «paymentStatus != PAID» — попадают ли cancelled брони с outstanding? Если да — будут фантомные просрочки по cancelled.

## Verdict

Spec в целом хорошо структурирован и грамотно опирается на upstream-отчёты. Главный архитектурный сдвиг (Booking-first + Invoice intermediate) — корректен. Но: 5 BLOCKER/HIGH-проблем (Phase 1 не закрывает то, что обещает; backfill искажает данные; нумерация без правил; WH-permission expansion не задокументирован; cancel-with-deposit edge case не описан) требуют доработки до того, как Agent 5 начнёт мокапить. Это не ошибки направления, это пропуски — но сумма пропусков такая, что мокап на текущей spec будет неполным.
