# Product Brief: Infrastructure Hardening

## Product Summary

**Что строим:**
- API key аутентификация для всех 40+ эндпоинтов Express API
- Rate limiting (express-rate-limit)
- Извлечение чистых функций из booking scene (1000 LOC) + unit-тесты
- Shared crewCalculator package (устранение дублирования web/bot)
- API smoke tests (supertest) — валидация auth на всех группах роутов

**Какие проблемы решаем:**
- Все эндпоинты публично доступны — любой с IP VPS может модифицировать данные
- Booking scene — revenue-critical путь без единого теста
- crewCalculator дублирован и уже расходится между приложениями
- Нет интеграционных тестов API — рефакторинг (auth) может сломать что угодно незаметно

**НЕ в скоупе:**
- JWT, сессии, управление пользователями
- OAuth / SSO
- Модуляризация booking scene на отдельные файлы
- Тесты Telegraf scene transitions
- Frontend login UI

**Ключевые решения:**
- API key (не JWT) для обоих потребителей
- Массив валидных ключей с самого начала (ротация без downtime)
- Двухфазный деплой: warn mode → enforce mode
- Извлечение хелперов из booking.ts без изменения поведения

## Problem Statement

API прокатной компании полностью открыт в интернет — 40+ эндпоинтов без аутентификации. Booking scene (основной путь заказа) не покрыт тестами. Crew calculator дублирован между двумя приложениями.

## Jobs to be Done

- Когда я деплою API на VPS, я хочу быть уверен, что только мои приложения имеют доступ к эндпоинтам, чтобы никто не мог повредить данные.
- Когда я меняю логику бронирования, я хочу иметь тесты, которые ловят регрессии, чтобы не сломать процесс заказа для клиентов.
- Когда я обновляю ставки в crew calculator, я хочу сделать это в одном месте, чтобы не забыть обновить вторую копию.

## User Stories

1. Как оператор, я хочу чтобы API отклонял запросы без валидного ключа, чтобы посторонние не могли читать/менять данные.
2. Как разработчик, я хочу чтобы бот и веб-прокси автоматически подставляли API key, чтобы ничего не ломалось после включения auth.
3. Как разработчик, я хочу unit-тесты для логики бронирования, чтобы безопасно менять booking scene.
4. Как разработчик, я хочу один пакет crewCalculator вместо двух копий, чтобы изменения применялись автоматически.
5. Как разработчик, я хочу smoke tests для API, чтобы убедиться, что auth работает на всех группах роутов.

## Success Criteria

- Все `/api/*` эндпоинты возвращают 401 без валидного API key
- Rate limiter блокирует >100 req/min с одного IP
- ≥80% покрытие чистых функций из booking scene
- crewCalculator импортируется из `@light-rental/shared` в обоих приложениях
- ≥1 smoke test на каждую из 12 групп роутов API

## Edge Cases

- Деплой API с auth раньше потребителей → warn mode предотвращает outage
- API key утечка → rate limiter + ротация через массив ключей
- Пустая корзина, дата начала = дата конца, 0 доступных единиц — покрыть тестами
