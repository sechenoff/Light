# Equipment Input: Catalog-First Redesign

**Date:** 2026-04-17
**Route:** `/bookings/new`
**Supersedes:** Two-mode pill-switcher from `2026-04-16-equipment-input-redesign`
**Mockup:** `docs/mockups/equipment-input-v5.html`

## Контекст

На `/bookings/new` сейчас работает двухрежимный pill-switcher — «AI ввод» и «Каталог» — которые пользователь переключает вручную. Кладовщик находит интерфейс неудобным: в любой момент видна только половина функциональности, и собрать комплект вручную, сверяясь с каталогом, невозможно без переключения контекста.

Редизайн убирает режимы и делает каталог единственным интерфейсом. AI-парсинг становится ускорителем: он предзаполняет выбор прямо в каталоге, а не создаёт параллельную таблицу.

## Цель

Дать кладовщику один экран, на котором он:
1. Видит весь каталог сразу (без аккордеонов, без раскрытия категорий).
2. Может собрать комплект вручную — кликом по строкам + степпер количества.
3. Может ускорить сборку через AI-вставку — гаффер-текст распознаётся и проставляет количества в каталоге.
4. Видит доступность, цены и итог в реальном времени.

## Архитектура секции «Оборудование»

```
┌─────────────────────────────────────────────────────────┐
│ Sticky header (остаётся при скролле):                    │
│   Заголовок «Оборудование»          · N позиций · SUM ₽ │
│   [Smart input: поиск / AI-вставка]          [AI-hint]  │
│   Табы: [Все·N] [Свет·3] [Камеры] [Звук] [Штативы·1]    │
├─────────────────────────────────────────────────────────┤
│ AI-баннеры (условно, после парсинга):                    │
│   ✔ Распознано 4 из 5                         [×]       │
│   ⚠ Не найдено: «Генератор 6кВт»   Добавить вручную    │
├─────────────────────────────────────────────────────────┤
│ Каталог (плоский, все категории раскрыты):               │
│   СВЕТ                                         3 выбрано │
│   ▌ARRI SkyPanel S60-C  4 000 ₽  [−][2][+]              │
│     ARRI SkyPanel S30-C  2 500 ₽  [+ Добавить]          │
│   ▌Kino Flo 4Bank 4ft    3 000 ₽  [−][1][+]             │
│     ...                                                  │
│   КАМЕРЫ                                                 │
│     ARRI ALEXA Mini LF   25 000 ₽ [+ Добавить]          │
│     ...                                                  │
├─────────────────────────────────────────────────────────┤
│ Footer: 4 позиции · 11 единиц          57 500 ₽         │
```

Три зоны:
- **Sticky-шапка** — название + smart-input + табы категорий. Прилипает к верху при скролле карточки, кладовщик всегда видит итог/поиск/табы.
- **Плоский каталог** — все позиции загружаются один раз при маунте. Категории — визуальные разделители (`<div class="cat-h">`), не `<details>`, не аккордеон. Подкатегорий первого уровня нет (если появятся — решается отдельным редизайном).
- **Футер** — счётчик выбранного и сумма за период.

- `EquipmentTable.tsx` — 3-уровневой таблицы resolved/needsReview/unmatched больше нет.
- `NeedsReviewRow.tsx` — больше не используется (top-1 AI-матч автоматически).
- `UnmatchedRow.tsx` — заменяется баннером `AiResultBanner` и off-catalog секцией.
- `CatalogBrowser.tsx` — логика переезжает в `CatalogList`.
- `CategoryAccordion.tsx` — сворачивание категорий не нужно.
- `QuickSearchBar.tsx` — функции уходят в `SmartInput`.
- `CatalogItemCard.tsx` — заменяется более компактным `CatalogRow`.

**Создаются:**
- `SmartInput.tsx` — textarea с автораспознаванием режима:
  - Однострочный ввод → debounced фильтр каталога по имени (300ms).
  - Многострочный/длинный ввод (>40 символов или есть `\n`) → появляется кнопка «Распознать».
- `CatalogList.tsx` — плоский список всех позиций, сгруппированных заголовками категорий. Принимает `items: AvailabilityRow[]`, `selected: Map<id, SelectionInfo>`, коллбэки.
- `CatalogRow.tsx` — одна строка: название + цена/день + доступность + action. Action = `+ Добавить` в обычном состоянии, степпер при выборе, пусто при `availableQuantity === 0`.
- `AiResultBanner.tsx` — баннеры «Распознано N из M» и «Не найдено» со списком unmatched позиций.

**Переиспользуются:**
- `SummaryPanel.tsx` — правая колонка. Расширяется мини-списком позиций (имя + `×qty`, без цен), читаем из `catalogSelected` и `offCatalogItems`.
- `ClientProjectCard.tsx`, `DatesCard.tsx`, `CommentCard.tsx` — без изменений.
- `types.ts` — чистится: удаляются `InputMode`, всё связанное с `EquipmentTableItem`-match-union остаётся (нужно для API-контракта).

**Переписывается:**
- `EquipmentCard.tsx` — становится тонкой обёрткой над `SmartInput` + `AiResultBanner` + `CatalogList` + footer. Prop `inputMode` и весь набор AI-props уходят внутрь. Новые props: `selected`, `offCatalogItems`, `onAdd`, `onQuantityChange`, `onRemove`, `onAddOffCatalog`, `pickupISO`, `returnISO`.

### Состояние в `page.tsx`

**Удаляется:**
- `inputMode: "ai" | "catalog"` — режима больше нет.
- `gafferText`, `gafferParsing`, `gafferError`, `parseResultCounts` — переезжают внутрь `EquipmentCard` (локальное состояние для AI-парсинга).
- `EquipmentTableItem[]` с match-union resolved/needsReview/unmatched — данные расплющиваются.

**Упрощается:**

Текущая структура:
```ts
items: EquipmentTableItem[]  // массив с match-discriminated-union
```

Новая структура:
```ts
// Позиции из каталога (с equipmentId)
catalogSelected: Map<equipmentId: string, { name, quantity, dailyPrice }>

// Позиции вне каталога (AI-unmatched, добавленные вручную)
offCatalogItems: Array<{ tempId: string, name: string, quantity: number }>

// Весь каталог с availability для выбранных дат
catalog: AvailabilityRow[]

// AI-результаты (для баннера)
unmatchedFromAi: string[]
```

Для `POST /api/bookings/draft`/`quote` формируется `items[]` объединением `catalogSelected` (с `equipmentId`) и `offCatalogItems` (без `equipmentId`, с `name`). Существующие контракты API уже поддерживают оба варианта через `EquipmentTableItem { kind: "resolved" | "unmatched" }` — используем как есть.

### Загрузка каталога

При монтировании `EquipmentCard` (или при смене дат) делается **один** вызов `GET /api/availability?start=<pickupISO>&end=<returnISO>` без фильтров. Ответ — весь каталог с `availableQuantity` на выбранный период.

Ограничения:
- Если позиций очень много (>500), в будущем можно добавить виртуализацию. Для MVP весь список в DOM.
- Фильтр по категории на табах — клиентский (из `catalog`).
- Фильтр по поиску — клиентский `includes` без учёта регистра. Не дёргает `/api/availability?search=` каждый раз.

### Поиск

- Ввод текста в `SmartInput` → `setSearchQuery(value)` → `CatalogList` фильтрует `catalog` по `name.toLowerCase().includes(query.toLowerCase())`.
- Если поиск не даёт результатов → показать пустое состояние «Ничего не найдено».
- Debounce 300ms — как сейчас в `QuickSearchBar`.

### AI-вставка

1. Пользователь вставляет/печатает многострочный текст в `SmartInput`.
2. Определение режима AI-вставки: текст содержит `\n` **или** длина > 40 символов → показывается кнопка «Распознать».
3. Клик на «Распознать» → `POST /api/bookings/parse-gaffer-review` с текущим текстом.
4. Ответ `{ resolved, needsReview, unmatched }`:
   - **resolved** — прямое совпадение, `catalogSelected.set(equipmentId, { name, quantity, dailyPrice })`.
   - **needsReview** — AI нашёл несколько кандидатов. Берётся первый (top-1), добавляется как `resolved`. Упрощение относительно текущей логики: вместо inline-выбора кандидата пользователь видит позицию в каталоге и при желании правит вручную. (*Если при тестах выяснится, что точность top-1 ниже 80%, вернём логику отдельным экраном.*)
   - **unmatched** — складываются в `unmatchedFromAi: string[]` и показываются в amber-баннере.
5. `SmartInput` свернулся до одной строки с текстом (disabled); справа — кнопка «Очистить», которая сбрасывает `unmatchedFromAi` и `gafferText`, возвращает textarea в редактируемое состояние, но **не снимает выбор в каталоге**.
6. Баннер «Распознано N из M» — закрываемый (кнопка `×`). Баннер «Не найдено» — не закрывается автоматически, пока пользователь либо добавит unmatched-строку в off-catalog, либо нажмёт «Игнорировать» (скрывает баннер, но строка остаётся в unmatched-массиве в памяти до новой вставки).

**Повторный парсинг**: если пользователь стирает disabled-текст и вводит новый → поведение как первый раз. Новые AI-совпадения мерджатся с текущим `catalogSelected` по `equipmentId` — если позиция уже выбрана, её количество перезаписывается значением из AI.

### Добавление «вручную» для unmatched (off-catalog позиции)

Клик на «Добавить вручную» рядом с unmatched-строкой:
1. Сразу создаёт off-catalog позицию: `{ tempId: uuid(), name: <unmatched-строка>, quantity: 1 }`.
2. Убирает эту строку из `unmatchedFromAi` → баннер пересчитывается.
3. Позиция появляется в отдельной секции **вверху каталога** под заголовком «Дополнительные позиции» — визуально как обычная `CatalogRow`, но без блока доступности и цены за день.
4. Эта секция отрисовывается только если `offCatalogItems.length > 0`.
5. Управление: степпер количества и кнопка удаления (корзина, `aria-label="Удалить позицию"`).

Серверная сторона: существующая логика парсинга `items[]` принимает записи без `equipmentId` (с `name`) и трактует их как off-catalog — используем как есть.

### Состояния строки каталога

1. **Обычная** (`availableQuantity > 0`, не выбрана):
   - Белый фон, hover → `bg-surface-muted`.
   - Справа: кнопка `+ Добавить` с `border-accent-border`, `text-accent-bright`.
2. **Выбранная** (есть в `catalogSelected`):
   - Фон `bg-green-bg` (#f0fdf4), зелёная полоска 3px слева (border-left).
   - Имя в `text-green` (var(--green)).
   - Справа: `[−][q][+]` степпер с зелёной рамкой.
   - Максимальное количество = `availableQuantity`. Кнопка `+` дизейблится на максимуме.
3. **Недоступная** (`availableQuantity === 0`, не выбрана):
   - opacity 0.35, pointer-events: none.
   - Кнопки «Добавить» нет — вместо неё текст `нет в наличии`.
4. **Выбранная + доступность упала** (была в `catalogSelected`, теперь `newAvailable < currentQty`):
   - Если `newAvailable > 0` → количество подрезается, amber-иконка + текст «скорректировано до N из M».
   - Если `newAvailable === 0` → фон `var(--rose-soft)`, rose-иконка ⚠ + «недоступно на новые даты», кнопка удаления активна, степпер скрыт.

### Смена дат после выбора

Если пользователь меняет даты `pickupISO`/`returnISO` после того, как уже выбрал позиции:
1. `EquipmentCard` делает новый `GET /api/availability?start&end` — каталог перезагружается.
2. `catalogSelected` остаётся как есть, но для каждой выбранной позиции проверяется `availableQuantity` из нового ответа:
   - Если `newAvailable >= currentQty` → без изменений.
   - Если `newAvailable < currentQty` → количество обрезается до `newAvailable`, строка помечается amber-индикатором «количество скорректировано».
   - Если `newAvailable === 0` → позиция остаётся в `catalogSelected` (не сбрасываем выбор без подтверждения), строка помечается rose-индикатором «недоступно на новые даты», пользователь видит и может убрать.
3. `offCatalogItems` на даты не завязаны — остаются без изменений.

### Табы категорий

- Считываются из `catalog` по полю `category`.
- Плюс фиксированный таб «Все» — показывает все категории.
- Счётчик выбранного в табе (`Свет · 3`) — в зелёном моноширинном шрифте.
- Клик на таб → `setActiveTab(name)` → `CatalogList` рендерит только позиции с `category === activeTab`.
- Поиск и таб работают вместе (AND).

## Изменения API

Никаких новых endpoints. Используются существующие:
- `GET /api/availability?start&end` — без `category` и `search`, клиент забирает всё.
- `POST /api/bookings/parse-gaffer-review` — без изменений.
- `POST /api/bookings/quote`, `POST /api/bookings/draft`, `POST /api/bookings/:id/submit-for-approval` — без изменений контракта. Меняется только форма, в которой клиент строит `items[]`.

## Прочие изменения интерфейса `/bookings/new`

Компактные верхние карточки:
- `ClientProjectCard` и `DatesCard` в мокапе v5 отображаются как `meta-card` — 2 колонки в ряд, только метки и значения (без формы редактирования в этом контейнере). **Вне scope текущего редизайна** — оставляем существующие `ClientProjectCard` и `DatesCard` как есть. Чтобы не раздувать задачу, менять их не будем. Если после ревью v5 решим упростить — отдельный спек.

`SummaryPanel` справа:
- Sticky — уже так и есть. Не меняем.
- Мини-список позиций в сайдбаре (в v5 есть) — **добавим**, но по упрощённой схеме: имя + `×qty`, читаем из `catalogSelected` и `offCatalogItems`. Без цен (цены считает quote-endpoint, показываем только в футере секции «Оборудование»).

- `AiResultBanner` — рендер ok/warn, закрытие success-баннера.

**Интеграционные тесты** (через существующий harness в `apps/web/src/components/bookings/__tests__/`):

  - Фильтр по поиску.
  - Фильтр по табу.
  - Добавление позиции → `onCatalogAdd` вызывается с правильным объектом.
  - Изменение количества через степпер → `onCatalogQuantityChange` с +1/-1.
  - Мок `POST /api/bookings/parse-gaffer-review` → AI-резолв, зелёные строки, warn-баннер для unmatched.
  - Кнопка «Очистить» — сбрасывает AI-текст, не снимает выбор.

**Нет регрессий:**
- Существующие тесты `apps/api/src/__tests__/` (approval.test.ts, dashboard.test.ts и т. д.) не затрагиваются — бэкенд не меняется.
- Тест `apps/bot/` (booking-helpers.test.ts) не затрагивается.

**Ручная проверка на дев-сервере:**
- Пустая бронь → каталог загружается, все категории видны, поиск работает.
- Вручную: клик «Добавить» → зелёная строка + степпер, изменение количества, удаление.
- AI: вставка реального гаффер-списка → зелёные строки совпадают с ожидаемыми позициями, unmatched показывается в баннере.
- Недоступные позиции (`availableQuantity === 0`) затемнены, кнопки «Добавить» нет.
- Сохранение черновика → валидный `POST /api/bookings/draft` с корректным `items[]`.

## Решения

- **Поиск и фильтр — клиентские**: каталог влезает в память (текущий `/api/availability` возвращает всё при отсутствии фильтров). Если в проде каталог вырастет >500 позиций, добавим серверный фильтр отдельной задачей.
- **NeedsReview сворачивается в top-1 AI-матч**: упрощает UI. Пользователь всегда может скорректировать в каталоге. Если метрика точности окажется ниже ожидаемой — возвращаем промежуточный шаг.
- **Подкатегории не вводим**: текущие категории плоские. Иерархия — отдельный проект.
- **Количество ограничено `availableQuantity`**: как и сейчас. Превышение блокируется на клиенте (UI) и на сервере (существующая проверка при `POST /api/bookings/quote`).
- **Состояние AI-парсинга локально в `EquipmentCard`**: `page.tsx` не должен знать про AI-текст. Это поможет в будущем добавить другие источники (импорт, шаблоны) без разрастания состояния страницы.

## Точки роста (вне scope MVP)

- Шаблоны комплектов («свет на интервью», «базовый camera-pack») — рядом с `SmartInput`.
- История предыдущих броней того же клиента — кнопка «повторить прошлую бронь».
- Виртуализация списка если каталог > 500 позиций.
- Перенос `ClientProjectCard` и `DatesCard` в формат `meta-card` из v5 — чисто косметика.
- Мини-список в `SummaryPanel` с ценами и удалением — в v5 есть без цен/удаления.
