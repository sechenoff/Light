Матрица прав — кто что видит и делает

Три роли: Руководитель, Кладовщик, Техник. Логика — каждый видит ровно столько, сколько нужно для его ежедневной работы. Минимум опций в боковом меню → меньше когнитивной нагрузки + проще обучать новых сотрудников.

Ниже полная матрица по разделам + легенда + обсуждение спорных мест (кто удаляет, кто редактирует задним числом, кто отменяет, кто принимает возврат и т. д.).

Спорные места — где важно договориться на берегу

Сценарий 1
Кладовщик может отменить бронь после «Одобрено»?
Только до выдачи. Если оборудование ещё на складе и клиент звонит с отменой — кладовщик может сам вернуть бронь в «Отменено» (убирает из слотов доступности). После выдачи отмена = возврат с пересчётом → нужна резолюция руководителя.
Сценарий 2
Кто заводит карточку ремонта?
Кладовщик — автоматически, когда на возврате нажимает «поломка». Система создаёт Repair-карточку со статусом «Ждёт ремонта», единицу переводит из «Выдано» в «Ждёт ремонта». Техник потом берёт в работу. Техник тоже может создать карточку вручную (плановая профилактика).
Сценарий 3
Техник видит, от какой брони пришла единица?
Видит номер и даты брони (чтобы понять контекст использования), но не видит клиента, цену, суммы. На карточке ремонта: «Arri Fresnel 2kW №4 · из брони № МФ-2026-03-15, даты 5–12 марта · причина: треснул рефлектор».
Сценарий 4
Кладовщик видит финансовые цифры?
Внутри брони — да (сумма аренды, залог, статус оплаты — нужно для разговора с клиентом). В карточке клиента — видит «общий долг» (чтобы не выдать оборудование должнику). Раздел /finance не видит вообще — ни в меню, ни по прямой ссылке.
Сценарий 5
Редактирование задним числом
Закрытая бронь (статус «Возвращено» или «Отменено») — только руководитель. Любое изменение пишется в аудит-лог (кто, когда, что было, что стало). Кладовщик — только через руководителя.
Сценарий 6
Прямая ссылка, если роль не видит страницу
HTTP 403 на уровне API (apiKeyAuth + rolesGuard middleware) + редирект на /day на фронте. В меню пункт вообще скрыт. Попытка прямого перехода = сразу «нет доступа, вернуться на главную».
Сценарий 7
Один человек совмещает роли?
У одного пользователя одна роль. Если руководитель хочет «побыть кладовщиком» — работает через Super-admin (у него все права кладовщика и сверху). Мультироли в этой итерации не делаем — усложняет матрицу и тестирование.
Сценарий 8
Что видно на «Моём дне»?
Все три роли видят /day, но содержимое зависит от роли. Руководитель: согласования + долги + аномалии расходов. Кладовщик: выдачи / возвраты / конфликты сегодняшнего дня. Техник: очередь ремонта + просроченные по SLA.

Техническая реализация

  • Prisma-схема. В модели User уже есть role: UserRole (сейчас SUPER_ADMIN, WAREHOUSE). Добавить TECHNICIAN.
  • Middleware rolesGuard — новый, в apps/api/src/middleware/rolesGuard.ts. Принимает массив разрешённых ролей, читает req.user.role, возвращает 403 {code: "FORBIDDEN_BY_ROLE"}. Навешивается на роуты в routes/index.ts.
  • Фронт — сокрытие меню. В apps/web/src/components/AppShell.tsx фильтруем массив пунктов по currentUser.role. При попытке прямого URL — useEffect-редирект на /day, если нет прав.
  • Аудит-лог — новая таблица AuditEntry (id, userId, action, entityType, entityId, before (JSON), after (JSON), createdAt). Пишется через сервисный слой на UPDATE/DELETE, не через Prisma middleware — чтобы не дублировать.
  • Тесты. Для каждой роли — набор smoke-тестов: «Кладовщик → GET /api/finance/debts → 403», «Техник → POST /api/bookings → 403», «Руководитель → всё 200». Минимум 20 кейсов на матрицу.
  • Существующий botScopeGuard (ограничивает ключи openclaw-*) остаётся как есть — это слой для ботов, поверх роли. Telegram-бот гафферов работает от имени «виртуального» пользователя-кладовщика.
